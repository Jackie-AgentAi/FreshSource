# 轻量级系统架构设计（MVP）

> **业务定位**：B2B 生鲜订货平台（订货方批量/稳定采购，店铺/供应商接单履约；数据模型与接口可与「买家/卖家」角色一一对应）  
> **架构阶段**：**单体应用**（不拆微服务、不引入消息总线；后续若增长再演进）  
> **依据文档**：`docs/requirement.md`、`CLAUDE.md`  
> **版本**：v1.0  
> **日期**：2026-04-13  

---

## 1. 系统整体架构说明

- **三端**：订货端（React Native + Expo）、履约端（RN + Expo，可与「卖家端」同一工程多角色或分壳）、管理后台（React + Ant Design Pro）。
- **服务端**：单进程 Go（Gin）承载全部业务；**MySQL** 持久化；**Redis** 做会话/限流/热点缓存（与 `CLAUDE.md` 一致）；静态与上传文件本地目录（预留 OSS）。
- **边界**：网关能力以内嵌中间件形式存在于同一进程（JWT、CORS、限流、日志），**不单独部署网关服务**。
- **非目标（MVP）**：独立支付中台、仓储 WMS、配送调度微服务、多租户独立库。

## 2. 后端模块划分（模块边界）

按 **包/目录边界** 划分，进程内通过 **Service 显式调用**，禁止循环依赖（下层不依赖上层 Handler）。

| 模块 | 包建议（对应 `internal/`） | 边界（负责 / 不负责） |
|------|---------------------------|------------------------|
| **auth** | `service/auth` + `handler/...` 鉴权入口 | 登录注册、JWT 双 Token、刷新登出；不负责业务权限细则（由各模块校验角色） |
| **user** | `model` + `repository/user` + `service/user` | 用户资料、角色、地址；不直接改订单与库存 |
| **shop** | `service/shop` | 店铺入驻、审核联动状态、店铺信息；不计算订单金额 |
| **catalog** | `service/catalog`（可与 product 合并为小模块） | 分类、商品展示字段、搜索列表；不写订单行 |
| **product** | `service/product` + `repository/product` | 商品 CRUD、审核状态、价格、起购量；**库存字段归属本模块表，但扣减逻辑由 order+inventory 协同** |
| **inventory** | `service/inventory`（可与 `product` 同包不同文件，但接口要清晰） | **库存查询、下单锁定、取消/超时释放、发货扣减语义**；不创建订单主单 |
| **cart** | `service/cart` | 购物车规则（按店分组、上限 100、失效标记）；下单时调用 order，不直接锁库存 |
| **order** | `service/order` | 订单创建、状态机驱动、订单日志 `order_logs`、运费计算（读系统配置）；**通过 inventory 完成锁/释库存** |
| **delivery**（履约） | `service/delivery` 或 `order` 内子过程（MVP 推荐：**先放在 order 内函数，接口层单独归类**，见第 7 节） | 接单、发货、送达、配送备注；不处理对账收款 |
| **finance**（对账） | `service/finance`（MVP 可极薄） | **线下/货到付款场景**：应收统计、导出、可选「对账标记」字段或简单对账记录表；**不接支付渠道** |
| **notification** | `service/notification` | 写通知表、供各端轮询；各业务模块**只调 notification 接口，不读对方表** |
| **review** | `service/review` | 评价与回复、店铺评分加权；依赖订单已完成状态 |
| **admin_config** | `service/config` + 后台 handler | 系统配置键（运费、自动取消分钟数等）；供 order 读取 |
| **common** | `pkg`、上传、短信等 | 无业务状态机 |

**依赖方向（摘要）**：`order` → `inventory`、`product`（校验）、`shop`、`user/address`；`cart` → `product`（校验）；`review` → `order`；`notification` ← 多模块；`finance` → `order`（只读聚合为主）。

## 3. 各模块职责说明

- **order**：维护订单主表/明细、状态流转、与 `order_logs` 一致；触发通知；调用 inventory 完成锁库/释放。
- **inventory**：保证**并发下库存一致**（事务 + 行级更新或等价手段）；对外提供「锁、释、发货扣减」等原子能力。
- **delivery（履约）**：在 MVP 中与订单强相关：卖家确认、标记配送、确认送达；**不引入独立配送员账号时，即卖家端接口能力**。
- **finance（对账）**：无在线支付前提下，聚焦**订单金额汇总、导出、账期/回款标记（可选）**；不做复杂会计科目。
- **product / catalog**：商品与分类、审核流；库存为 0 自动下架等规则在 product+inventory 内协调。
- **shop**：店铺审核、营业状态；订单校验店铺是否可售。
- **user / auth**：身份与地址；JWT 解析在 middleware，业务层只拿 `user_id` + 角色。

## 4. 核心业务流程（下单 / 发货 / 对账）

### 4.1 下单（订货方）

1. 购物车校验：商品上架、库存≥购买量、起购量、按店分组。
2. 确认页：计算商品小计、运费（`delivery_base_fee` / `delivery_free_threshold`）。
3. **创建订单**（同店一单、多店多订单）：事务内创建 `orders` / `order_items`，**调用 inventory 锁定库存**，写 `order_logs`，初始状态 **0 待确认**。
4. 超时任务（定时或延迟队列简化版：**定时扫表**即可 MVP）：待确认超过配置分钟 → 取消订单、释放库存、记日志、通知。

### 4.2 发货（履约方 / 卖家）

1. **待确认 → 已确认**：卖家确认接单；或拒绝/取消路径进入 **已取消** 并释放库存（与需求一致）。
2. **已确认 → 配送中**：卖家标记发货/配送。
3. **配送中 → 已送达**：卖家或流程确认到达。
4. **已送达 → 已完成**：买家确认收货，或超过配置小时自动完成。
5. 全程写 `order_logs`，关键节点发 **notification**。

### 4.3 对账（线下结算 MVP）

1. **数据源**：`orders` 中 `pay_amount`、`status`（已完成视为待回款/已结清由业务标记决定）。
2. **能力建议**：管理端按店铺+时间筛选导出；可选字段 `settlement_status`（未对账/已核对）— 若一期未建表，可用运营线下表格，**代码预留接口与查询条件即可**。
3. **原则**：不与订单状态机耦合；**不改变 0–7 订单状态语义**。

## 5. 订单状态流转设计（完整）

状态编码与库表注释一致：`0 待确认 1 已确认 2 配送中 3 已送达 4 已完成 5 已取消 6 退货中 7 已退货`。

### 5.1 主流程（正常履约）

| 自状态 | 到状态 | 触发方 / 条件 |
|--------|--------|----------------|
| — | 0 待确认 | 订货方下单成功（已锁库存） |
| 0 | 1 已确认 | 卖家确认接单 |
| 0 | 5 已取消 | 买家取消 / 卖家拒绝 / 系统超时；**释放库存** |
| 1 | 2 配送中 | 卖家标记配送 |
| 1 | 5 已取消 | MVP：**若允许发货前卖家取消**，需释放库存（与产品确认；需求默认重点在待确认取消，此处保留为可配置扩展） |
| 2 | 3 已送达 | 卖家确认送达 |
| 3 | 4 已完成 | 买家确认收货 **或** 超时自动完成 |
| 3 | 5 已取消 | **一般不推荐**；若业务禁止，可在服务层直接拒绝 |

### 5.2 退货扩展（表已支持 6/7）

MVP **若不开放买家自助退货**，仍应在设计层写全流转，避免后续改状态机冲突：

| 自状态 | 到状态 | 说明 |
|--------|--------|------|
| 4 已完成 | 6 退货中 | 发起退货（管理员仲裁或线下协商后由 **admin** 标记；或二期买家申请） |
| 6 退货中 | 7 已退货 | 退货完成确认（**admin** 或卖家确认，以产品规则为准） |
| 6 | 4 | 驳回退货，回到已完成（可选，需产品确认；实现时统一走 `order_logs`） |

**终点状态**：`5 已取消`、`4 已完成`、`7 已退货` 不再进入正常配送链路；`4` 可进入 `6`。

### 5.3 不允许的跳转（实现时校验）

- 任意状态 **不可回退到 0**。
- **已完成 / 已退货 / 已取消** 不可再转为配送中（除非单独做「异常纠偏」且仅 admin，并强制记日志，MVP 建议不做隐式回退）。

## 6. 模块之间调用关系（谁可以调用谁）

```
                    ┌─────────────┐
                    │ notification│
                    └▲───▲───▲───┘
                     │   │   │
     ┌───────────────┘   │   └────────────────┐
     │                   │                    │
┌────┴────┐        ┌─────┴─────┐        ┌─────┴────┐
│  order  │◄───────│ inventory│        │  review  │
└────┬────┘        └─────▲─────┘        └────▲─────┘
     │                 │                   │
     │ reads           │ locks stock       │ reads
     ▼                 │                   │
┌────┴────┐      ┌─────┴─────┐      ┌──────┴─────┐
│  shop   │      │  product  │      │   order    │
└─────────┘      └───────────┘      └────────────┘

┌─────────┐      只读订单汇总/导出
│ finance │ ─────────────────► order（及明细）
└─────────┘

cart ──► product（校验）, 下单时 ──► order
user / auth：被各 handler 使用；service 层通过参数传入 buyer_id / seller_id
admin：运维审核 shop/product，可纠偏订单（谨慎、记日志）
```

**规则**：

- **inventory** 仅被 **order**（及明确的 admin 库存修正，若有）调用；cart **不**直接锁库存。
- **finance** **禁止**调用 inventory；**允许**读 order 聚合数据。
- **review** 只依赖 **order** 已完成等可读状态，不回调改订单金额。

## 7. API 分层规范（admin / customer / delivery）

与实现路径映射（与 `docs/requirement.md`、`CLAUDE.md` **保持一致**，避免两套前缀）：

| 架构分层名称 | 使用者 | HTTP 前缀（实现） | 说明 |
|--------------|--------|-------------------|------|
| **admin** | 平台运营 | `/api/v1/admin/` | 审核、配置、监控、导出、可选退货仲裁 |
| **customer** | 订货方（B2B 采购方） | `/api/v1/buyer/` | 商品浏览、购物车、下单、订单查看与收货 |
| **delivery** | 履约方（供应商/卖家；未来可扩展专职配送） | `/api/v1/seller/` 中 **订单履约**相关接口 | MVP：**不新增**独立 `/delivery/` 服务进程；架构文档称 delivery 层，**代码落在 seller 路由**（确认接单、发货、送达）。若未来有独立配送 App，再增加 `/api/v1/delivery/` 仅承载配送员动作，仍属同一单体仓库。 |
| **common** | 三端共用 | `/api/v1/common/` | 认证、上传、地区等 |

**统一约定**：JSON；成功 `code: 0`；错误码区间见需求文档 8.2（系统/用户/商品/订单/店铺）。

## 8. 技术架构图（文本描述）

**客户端**：订货 App、履约 App（卖家）、Admin Web 通过 HTTPS 访问 **Nginx**（静态资源 + 反向代理）。

**服务端（单实例/可水平扩展多实例，共享 MySQL）**：

- **Gin**：路由分组 `buyer` / `seller` / `admin` / `common`；中间件 JWT、CORS、限流、日志（Zap）。
- **业务层**：各 `service` 模块（见第 2 节）；**无 RPC、无独立订单微服务**。
- **数据**：MySQL（GORM）；Redis（缓存与限流等）；本地 `/uploads`（图片与凭证）。

**异步与任务**：MVP 用 **定时任务**（如每分钟扫描超时未确认订单）即可；不强制引入 Kafka/RabbitMQ。

**部署**：Docker Compose：应用 + MySQL + Redis + Nginx（与团队现状对齐即可）。

## 10. MVP 落地检查清单（给开发用）

- [ ] 订单状态机与 `order_logs`、库存锁释一致  
- [ ] `customer(buyer)` / `delivery(seller履约)` / `admin` 路由分离，中间件校验角色  
- [ ] 系统配置键：运费、自动取消分钟数、自动完成小时数（与需求一致）  
- [ ] 对账：至少支持管理端按时间与店铺导出订单；财务扩展字段按需加表  
- [ ] 状态 6/7：服务层校验合法迁移；若一期无前端入口，仅 admin 可操作并记日志
