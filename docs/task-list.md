## 核心业务规则：订单 / 库存 / 账务（实现必须一致）

本节为 **规范级** 说明，与 `docs/db-design.md`、`docs/architecture.md` 一致；若冲突以 **本清单 + db-design** 联判。

### 1）创建订单时：锁库存还是扣减？

- **MVP 采用「事务内直接扣减 `products.stock`」**（`UPDATE ... WHERE stock >= quantity`）；**不增加 `locked_stock` 列**。
- **语义**：扣减成功即视为该部分库存已被本订单占用；与「先锁后扣」在数据上等价，便于少一张状态表。
- **创建失败**：任一明细扣减失败则**整单回滚**，不生成订单行。

### 2）何时释放库存？如何回滚？

| 场景 | 订单变化 | 库存 |
|------|----------|------|
| 订货方取消 | `0 → 5`（仅允许 status=0） | 按 `order_items` **加回** `products.stock` |
| 发货方拒单 | `0 → 5` | **加回** |
| 系统超时取消 | `0 → 5`（定时任务） | **加回** |
| 其它状态 → 5 | MVP **仅允许上述路径**产生释放；若扩展「1→5 发货前取消」须单独开任务并补验收 | 若扩展须定义是否加回（默认须加回） |

释放与写 `order_logs`、`notifications` 应在**同一业务事务或明确失败补偿策略**内可追溯。

### 3）签收、完成、对账标记三者的关系

| 概念 | 含义 | 谁操作 / 触发 |
|------|------|----------------|
| **已送达 (3)** | 履约节点：货已到约定交付点 | 发货端 `PUT .../arrived`（seller，`1→2→3` 链路中 2→3） |
| **已完成 (4)** | 订单履约闭环，可评价 | 订货端 `PUT .../receive`（`3→4`）**或** 定时任务：已送达超过配置小时自动 `3→4` |
| **对账标记** `orders.settlement_status` | **线下回款核对标记**，不参与状态机 | **仅管理端** `PUT .../settlement`；与 0–7 **解耦** |

- **无支付**：不存在「支付成功」事件；**`pay_amount` 为下单时快照，不在后续流程中因支付而改**。
- **应收（运营口径）**：以「已完成订单」的 `pay_amount` 作为对账依据即可；**是否回款只看线下 + `settlement_status`**。

### 4）默认配置键（验收与演示必须可读）

| config_key | 默认值（与需求一致） | 用途 |
|------------|----------------------|------|
| `order_auto_cancel_minutes` | `30` | 待确认超时 → 取消 + 释放库存 |
| `order_auto_complete_hours` | `24` | 已送达超时 → 自动确认收货 → 已完成 |
| `delivery_base_fee` | `5.00` | 未达包邮门槛时运费 |
| `delivery_free_threshold` | `50.00` | 商品金额 ≥ 此值则 `freight_amount=0` |

**定时任务**须 **读取库内配置**；改配置后新单与新调度生效（MVP 可不实现「已生成订单重算」）。

### 5）多店拆单

- **必须**：购物车含多个 `shop_id` 时，**一次提交生成多张订单**（每张对应一个店铺），**多张订单各自独立 `order_no`、独立状态机**。
- **验收**：同一次 `POST /buyer/orders` 返回多个 `order_id`。

### 6）库存不足

- HTTP 业务失败返回 **`code = 30002`**（与 `docs/requirement.md` 8.2 一致）；**不得**创建部分订单。

### 7）状态迁移权限（写死）

| 迁移 | 允许角色 | 接口归属（见 api-design） |
|------|----------|---------------------------|
| `0 → 5` 买家取消 | 订货端，且 `buyer_id` 匹配 | buyer `.../cancel` |
| `0 → 5` 卖家拒单 | 发货端，且 `seller_id` 匹配 | seller `.../reject` |
| `0 → 5` 超时 | 系统 `operator_role=4` | 定时任务，无 HTTP |
| `0 → 1` | 发货端卖家 | seller `.../confirm` |
| `1 → 2` | 发货端卖家 | seller `.../deliver` |
| `2 → 3` | 发货端卖家 | seller `.../arrived` |
| `3 → 4` | 订货端本人 **或** 系统超时 | buyer `.../receive` / 定时任务 |
| `4 → 6 → 7`（退货） | MVP **仅管理端**（若启用） | admin `.../status`；禁止买家自助则不在 buyer 暴露 |
| 非法迁移 | - | 统一 **`code = 40002`** |

**admin** 默认 **不得**绕过规则把订单从 5/7 改回进行中（纠偏单开任务）。

## MVP 上线闸门：管理后台最小可用范围（强制）

下列能力在 **对外的 MVP 演示 / 试用 / 预发** 之前 **必须可用**（界面可极简，但须**可点击完成闭环**）：

1. **商品审核**（通过/驳回，商品从「审核中」到可上架逻辑可跑通）  
2. **店铺审核**（否则卖家店铺 `audit_status≠1` 时订货端应拒单，链路堵死）  
3. **订单查询**（列表 + 详情即可，用于运营与对账联调）  
4. **配置项维护**（至少能改运费、超时分钟/小时，否则无法验收定时任务）

**排期约束**：对应后端任务以 `P2-1-*` 编号为主，**允许且建议与 P1 中后期并行**，**禁止**团队默认「等 P2 最后一两周再做后台」导致 P1 联调长期依赖手工改库。

**映射**：`P2-1-2`（店铺）`P2-1-3`（商品）`P2-1-4`（订单列表/详情，导出可紧随）`P2-1-5`（配置 + 轮播可后移但**配置优先**）+ `P2-2-2` 中上述四块**最小页面**。

## P1 核心链路验收规则（可测试条件汇总）

以下为 **P1 完成定义（DoD）** 的硬条件；与上节「状态权限」一致处不重复赘述。

| 编号 | 验收条件 |
|------|----------|
| AC-1 | 库存不足时 `POST /buyer/orders` 返回 **`code=30002`**，DB 中 **`orders` 无新行**、`products.stock` **不变** |
| AC-2 | 多店铺结算时 **单次创建请求** 返回 **≥2 个 `order_id`**，且各单 `shop_id` 唯一对应 |
| AC-3 | `order_auto_cancel_minutes`（默认 **30**）内未接单，订单 **`0→5`**，`order_logs` 含 `to_status=5` 且 `operator_role=4`，库存 **已加回** |
| AC-4 | 已送达后超过 `order_auto_complete_hours`（默认 **24**），订单 **`3→4`**，无需人工点击（可通过改系统时间或临时改配置为更小值测） |
| AC-5 | 订货端 **仅** `status=0` 可 `cancel`；发货端 **仅** `0` 可 `reject`、`0` 可 `confirm`、`1` 可 `deliver`、`2` 可 `arrived`；越权一律 **`40002` 或 `10003`**（团队二选一固定） |
| AC-6 | 运费：商品小计 **≥** `delivery_free_threshold` 时 **`freight_amount=0`**；否则 **`freight_amount`** 等于 `delivery_base_fee`（单笔订单维度） |
| AC-7 | 店铺 `audit_status≠1` 或营业关闭时，下单接口返回 **`50001`** 或业务约定码，**且不建单** |
| AC-8 | 任一合法状态迁移后 **`order_logs` 必有一行**（`from_status`/`to_status` 正确） |
