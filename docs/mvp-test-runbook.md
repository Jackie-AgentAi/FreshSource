# FreshMart MVP 测试排差执行手册

> 适用阶段：全链路联调验收、真机/模拟器走查、MVP 交付前排差。  
> 依据：`docs/dev-spec.md` v1.3、`docs/task-list.md` v1.1、`docs/api-design.md` v1.1.9、`docs/db-design.md` v1.1、`docs/requirement.md`。  
> 范围：订货端 APP、发货端 APP、管理后台 Web、后端核心业务与生产化收口。  
> 明确不测：在线支付、退款、分账、IM、独立配送微服务、营销优惠券。

---

## 1. 测试目标与通过标准

### 1.1 本轮最重要目标

1. 真机/模拟器上完整点击走查订货端、发货端与管理后台。
2. 跑通主链路：买家下单 -> 卖家接单/拒单 -> 发货 -> 送达 -> 买家收货 -> 库存扣减/回滚 -> 消息/统计刷新。
3. 验证管理后台四个 MVP 闸门：店铺审核、商品审核、订单管理/对账、配置管理。
4. 补齐后端核心业务测试口径：库存事务、拒单回滚、非法状态流转、通知生成。
5. 标记生产化剩余风险：环境变量、上传策略、日志、鉴权、Expo 构建、Swagger。

### 1.2 MVP 可交付最低通过标准

| 编号 | 标准 | 通过条件 |
|------|------|----------|
| G-1 | 主链路闭环 | 买家可下单，卖家可接单、发货、送达，买家可收货完成 |
| G-2 | 库存正确 | 下单扣库存；买家取消、卖家拒单、超时取消均回滚库存 |
| G-3 | 状态机正确 | 合法迁移成功，非法迁移返回 `40002` 或统一约定的越权码 |
| G-4 | 后台闸门可用 | 店铺审核、商品审核、订单查询/对账、配置维护均可点击完成 |
| G-5 | 三端体验可用 | 登录、上传图片、列表刷新、底部安全区、键盘遮挡无阻断问题 |
| G-6 | 接口契约一致 | JSON 响应统一为 `code`、`message`、`data`；分页为 `list` + `pagination` |
| G-7 | 无越界功能 | 不出现支付、退款、分账、IM、独立配送端等 MVP 外能力 |

## 2. 测试环境准备

### 2.1 服务启动

仓库根目录：

```bash
docker compose up -d --build
```

服务地址：

| 服务 | 地址 |
|------|------|
| API | `http://localhost:8080` |
| Nginx 代理 | `http://localhost` |
| 管理后台 | `http://localhost:8000` |
| Swagger | `http://localhost/swagger/index.html` |
| 健康检查 | `GET http://localhost:8080/health` |

重建空库：

```bash
docker compose down -v
docker compose up -d --build
```

### 2.3 APP 启动

订货端：

```bash
cd buyer-app
npm install
npm run start
```

发货端：

```bash
cd seller-app
npm install
npm run start
```

端口约定：

| APP | Expo 端口 |
|-----|-----------|
| 订货端 | `8081` |
| 发货端 | `8082` |

真机测试时，手机与电脑需在同一局域网；若自动推导后端地址失败，设置 `EXPO_PUBLIC_API_BASE_URL` 指向电脑局域网 IP，例如 `http://192.168.1.10:8080`。

### 2.4 测试数据准备

如果新库只有账号和店铺，需先准备可售商品：

1. 用卖家账号登录发货端。
2. 发布至少 3 个商品：
   - 商品 A：库存充足，单价小于包邮门槛。
   - 商品 B：库存充足，和商品 A 不同分类。
   - 商品 C：低库存，例如库存 `1`，用于库存不足测试。
3. 用管理员登录管理后台。
4. 在商品审核中将商品 A/B/C 审核通过。
5. 确认商品 A/B/C 在订货端商品列表可见。

建议额外准备第二个卖家和店铺，用于多店拆单测试；若没有界面注册入口，可通过 API 或临时种子数据创建。

## 3. 缺陷记录格式

每个问题建议按以下模板记录，方便开发排查：

```text
标题：
环境：真机/模拟器/浏览器 + 系统版本 + APP 端
账号：
前置数据：
复现步骤：
实际结果：
期望结果：
接口信息：方法 + URL + request/response code/message
截图/录屏：
严重级别：Blocker / Critical / Major / Minor
是否回归通过：
```

严重级别建议：

| 级别 | 定义 |
|------|------|
| Blocker | 主链路无法继续、服务启动失败、登录不可用、下单不可用 |
| Critical | 数据错误、库存错误、状态机错误、越权、订单金额错误 |
| Major | 关键页面或关键按钮异常，但可绕行 |
| Minor | 文案、样式、空态、非阻断体验问题 |

## 5. 全链路业务验收

### 5.1 主路径：下单到完成

| 步骤 | 操作端 | 操作 | 期望/校验点 |
|------|--------|------|-------------|
| 1 | 管理后台 | 确认店铺已审核通过且营业中 | `audit_status=1`，`status=1` |
| 2 | 管理后台 | 确认商品已审核通过且库存 > 0 | 买家端可见 |
| 3 | 订货端 | 加购商品 A，进入购物车 | 商品按店铺分组 |
| 4 | 订货端 | 选择地址，提交订单确认 | 只返回预览，不创建订单 |
| 5 | 订货端 | 创建订单 | 返回 `order_ids`，订单状态为 `0=待确认` |
| 6 | 发货端 | 查看待确认订单 | 新订单可见，商品、地址、金额一致 |
| 7 | 发货端 | 点击接单 | 状态 `0 -> 1`，写 `order_logs` |
| 8 | 发货端 | 点击发货 | 状态 `1 -> 2`，写 `order_logs` |
| 9 | 发货端 | 点击送达 | 状态 `2 -> 3`，写 `order_logs`，`delivered_at` 有值 |
| 10 | 订货端 | 查看订单并确认收货 | 状态 `3 -> 4`，写 `order_logs`，`completed_at` 有值 |
| 11 | 管理后台 | 查看订单详情 | 状态、金额、买家/卖家信息与 APP 一致 |
| 12 | 管理后台 | 标记对账 | `settlement_status` 更新，不改变 `status` |

通过条件：

- 每次状态变化后三端刷新都一致。
- 所有响应保持 `{ code, message, data }`。
- 每个合法迁移都有 `order_logs`。
- 管理端对账只改 `settlement_status`，不改订单 `status`。

### 5.2 拒单回滚

| 步骤 | 操作 | 期望 |
|------|------|------|
| 1 | 记录商品 A 当前库存 | 记为 `stock_before` |
| 2 | 买家下单商品 A 数量 `q` | 创建成功后库存为 `stock_before - q` |
| 3 | 卖家拒单并填写原因 | 订单 `0 -> 5`，`cancel_by=2` |
| 4 | 刷新商品库存 | 库存恢复为 `stock_before` |
| 5 | 查看订单日志 | 有 `0 -> 5`，操作角色为卖家 |
| 6 | 买家查看订单 | 看到已取消/拒单原因 |

失败判定：

- 库存没有回滚：Critical。
- 拒单无原因也成功：Major 或 Critical，按接口约定应返回 `10001`。
- 拒单后还能接单/发货：Critical。

### 5.3 买家取消回滚

| 步骤 | 操作 | 期望 |
|------|------|------|
| 1 | 记录商品库存 | `stock_before` |
| 2 | 买家下单 | 库存扣减 |
| 3 | 买家在待确认状态取消 | 订单 `0 -> 5` |
| 4 | 刷新库存 | 库存回到 `stock_before` |
| 5 | 卖家端刷新 | 订单不再可接单 |

### 5.4 库存不足与禁止部分建单

| 步骤 | 操作 | 期望 |
|------|------|------|
| 1 | 准备低库存商品 C，库存为 `1` | 买家端可见 |
| 2 | 买家尝试下单数量大于库存 | `POST /buyer/orders` 返回 `code=30002` |
| 3 | 查询订单列表 | 没有新订单 |
| 4 | 查询商品库存 | 库存不变 |

如果购物车中有多个商品，其中一个库存不足，也不得创建部分订单。

### 5.5 多店拆单

前置：至少两个已审核营业店铺，各有可售商品。

| 步骤 | 操作 | 期望 |
|------|------|------|
| 1 | 买家分别加购两个店铺商品 | 购物车按店铺分组 |
| 2 | 进入确认订单 | 显示两个店铺分组，各自金额/运费 |
| 3 | 提交订单 | 单次请求返回至少 2 个 `order_id` |
| 4 | 买家订单列表 | 出现两张订单，`order_no` 不同 |
| 5 | 两个卖家分别登录 | 只能看到自己店铺的订单 |

### 5.6 非法状态流转

| 用例 | 操作 | 期望 |
|------|------|------|
| FSM-01 待确认直接发货 | 卖家对 `status=0` 订单点发货 | 返回 `40002`，状态不变 |
| FSM-02 已确认后买家取消 | 买家对 `status=1` 订单点取消 | 返回 `40002`，库存不回滚 |
| FSM-03 配送中确认收货 | 买家对 `status=2` 订单确认收货 | 返回 `40002` |
| FSM-04 已完成再次收货 | 买家对 `status=4` 订单确认收货 | 返回 `40002` |
| FSM-05 非归属订单操作 | 其他卖家或买家操作订单 | 返回 `10003` 或团队固定越权码 |

### 5.7 自动任务

#### 待确认超时取消

1. 管理后台将 `order_auto_cancel_minutes` 临时改为 `1`。
2. 买家创建待确认订单，不让卖家接单。
3. 等待定时任务下一轮执行。
4. 期望：
   - 订单 `0 -> 5`。
   - 库存加回。
   - `order_logs.operator_role=4`。
   - 买家/卖家列表刷新后状态一致。

#### 已送达自动完成

1. 管理后台将 `order_auto_complete_hours` 调小到便于测试的值。
2. 跑到订单 `status=3`。
3. 等待定时任务执行。
4. 期望：
   - 订单 `3 -> 4`。
   - 无需买家点击。
   - 写 `order_logs.operator_role=4`。

测试结束后恢复配置默认值：

| 配置项 | 默认值 |
|--------|--------|
| `order_auto_cancel_minutes` | `30` |
| `order_auto_complete_hours` | `24` |
| `delivery_base_fee` | `5.00` |
| `delivery_free_threshold` | `50.00` |
